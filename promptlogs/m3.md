## Entry 5
**Date:** May 18, 2026
**Task:** Fixing a Bug — USER Role Cannot Delete Customer Even With Permission Granted

**Prompt given to AI:** *"Why can't a user with user_role == 'user' delete a customer even when they are given the CUST_DEL permission? The error shown is: 'new row violates row-level security policy for table customers'."*

**What the AI produced:** AI identified the root cause: PostgreSQL checks the **new row state** against SELECT policies during UPDATE. After the soft-delete sets `record_status = 'INACTIVE'`, the updated row fails the existing `user_view_active_customers` SELECT policy (`USING (record_status = 'ACTIVE')`). Since no SELECT policy allowed USER-role accounts to see INACTIVE rows, PostgreSQL rejected the UPDATE entirely. The fix was a new SELECT policy: `user_cust_del_can_see_all_customers` scoped to users with `CUST_DEL` permission.

**What I changed / improved:** Ran the policy in the Supabase SQL Editor and confirmed the soft-delete operation succeeded. Also updated `AuthProvider.tsx` to add a realtime subscription on the `user_permission` table so that permission changes made by an admin are reflected in the target user's session immediately without requiring a re-login.

**Why:** Two separate layers were broken — the RLS layer blocked the DB write, and the app layer had no mechanism to pick up live permission changes for an active session.
